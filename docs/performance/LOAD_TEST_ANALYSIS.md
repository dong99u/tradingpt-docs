## 4. 원인 분석

### Root Cause (근본 원인)

#### 1차 원인: BCrypt 비밀번호 해싱 (Primary Bottleneck)

```java
// Spring Security의 BCrypt 인코더
// 기본 Cost Factor: 10 (2^10 = 1024 iterations)

@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(); // 기본값 strength = 10
}
```

- **CPU 집약적 연산**: BCrypt는 의도적으로 느리게 설계됨 (brute-force 방어)
- **단일 검증 시간**: 약 100-200ms의 CPU 연산 필요
- **500+ 동시 로그인**: EC2 CPU 96%까지 포화

#### 2차 원인: RDS 인스턴스 사양 제한

- **db.t4g.medium**: 단 2개의 vCPU
- **CPU 사용률 76%**: 쿼리 처리 지연 발생
- **Connection 증가**: 18 → 41 (요청 대기열 증가)

#### 3차 원인: EC2 T3 버스트 크레딧 고갈

- **T3 인스턴스 특성**: CPU 크레딧 기반 버스팅
- **지속적 고부하**: 크레딧 소진 시 baseline 성능으로 제한
- **Server #3 낮은 사용률**: 로드밸런싱 불균형 의심

### 5 Whys 분석

1. **Why 1**: 왜 응답 시간이 10초나 걸렸는가?
   - **Answer**: EC2 CPU가 96%까지 포화되어 요청 처리가 지연됨

2. **Why 2**: 왜 EC2 CPU가 포화 상태에 도달했는가?
   - **Answer**: 모든 VU가 매 iteration마다 로그인 API를 호출하여 BCrypt 연산 부하 집중

3. **Why 3**: 왜 BCrypt 연산이 CPU를 과도하게 사용하는가?
   - **Answer**: BCrypt는 보안을 위해 의도적으로 CPU 집약적으로 설계됨 (Cost Factor 10 = 2^10 iterations)

4. **Why 4**: 왜 현재 인프라가 이 부하를 감당하지 못하는가?
   - **Answer**: T3 인스턴스는 CPU 집약적 작업에 최적화되지 않음 + 버스트 크레딧 제한

5. **Why 5**: 왜 이런 인프라 구성을 선택했는가?
   - **Answer**: 초기 MVP 단계에서 비용 효율적인 T3 인스턴스 선택 + 부하 테스트 미실시

## 9. 면접 Q&A (Interview Questions)

### Q1. 이 문제를 어떻게 발견하고 분석했나요?

**A**: Inflearn 부하 테스트 강의를 수강하며 k6를 활용해 운영 서버에 대한 부하 테스트를 실행했습니다. 테스트 중 1000 VU 시나리오에서 응답 시간이 10초를 초과하고, TPS가 46.1/s에 불과한 것을 확인했습니다.

AWS CloudWatch를 통해 EC2 CPU 사용률이 96%까지 치솟고 RDS CPU도 76%에 도달한 것을 확인하여, CPU 바운드 병목임을 파악했습니다. 코드 레벨에서 분석한 결과 BCrypt 비밀번호 검증이 CPU 집약적 연산(100-200ms/요청)임을 확인하고, 이것이 1차 병목 원인임을 밝혀냈습니다.

**포인트**:
- k6를 활용한 체계적인 부하 테스트 수행
- CloudWatch 메트릭을 통한 인프라 레벨 모니터링
- 코드 레벨 분석으로 BCrypt 병목 식별

### Q2. 여러 해결 방안 중 최종 방안을 선택한 이유는?

**A**: 7가지 해결 방안을 비용, 복잡도, 보안 영향 측면에서 분석했습니다.

BCrypt Cost Factor 감소(10→8)는 즉시 적용 가능하지만 보안 약화(brute-force 취약성 2배)를 감수해야 합니다. 트레이딩 플랫폼의 특성상 보안이 중요하여 이 방안은 제외했습니다.

최종적으로 단기/중기/장기 3단계 접근법을 선택했습니다:
- **단기**: 테스트 시나리오 최적화로 현실적 베이스라인 확보
- **중기**: EC2 c5.large + RDS large 스케일업 (월 $146 추가)
- **장기**: Auto Scaling 도입으로 탄력적 인프라 구축

**포인트**:

### Q3. 이 문제의 기술적 근본 원인은 무엇인가요?

**A**: 근본 원인은 **BCrypt의 의도적 설계 특성**과 **인프라 사이징 미스매치**입니다.

BCrypt는 brute-force 공격 방어를 위해 의도적으로 느리게 설계되었습니다. Cost Factor 10은 2^10 = 1024번의 iteration을 수행하며, 단일 검증에 100-200ms의 CPU 시간이 소요됩니다.

T3 인스턴스는 버스트 기반 CPU 크레딧 시스템을 사용하여, BCrypt와 같은 지속적 CPU 집약 작업에 적합하지 않습니다. 또한 RDS db.t4g.medium의 2 vCPU는 동시 접속자 증가 시 병목이 됩니다.

결론적으로 CPU 집약적 인증 로직(BCrypt)과 버스트 기반 인프라(T3)의 조합이 근본 원인입니다.

**포인트**:
- BCrypt Cost Factor와 보안-성능 트레이드오프 이해
- AWS T3 인스턴스의 버스트 크레딧 메커니즘 이해
- 워크로드 특성에 맞는 인스턴스 타입 선택의 중요성

### Q4. 해결 과정에서 어떤 어려움이 있었고, 어떻게 극복했나요?

**A**: 주요 어려움은 다음과 같았습니다:

1. **운영 환경 테스트의 리스크**: 실제 운영 서버에서 부하 테스트 시 기존 사용자에게 영향을 줄 수 있었습니다. 이를 해결하기 위해 새벽 시간대(09:15-09:23)에 테스트를 수행하고, 별도의 테스트 계정 1000개를 생성하여 기존 데이터와 분리했습니다.

2. **복합적인 병목 원인 식별**: 초기에는 단순히 "서버가 느리다"는 현상만 보였습니다. CloudWatch에서 EC2/RDS의 CPU, 커넥션 수, 지연 시간 등 다양한 메트릭을 교차 분석하여 BCrypt가 1차 원인, RDS가 2차 원인임을 밝혀냈습니다.

3. **보안과 성능의 균형**: BCrypt Cost를 낮추면 즉시 해결되지만 보안 약화가 우려되었습니다. OWASP 권장사항과 업계 표준을 조사하여 인프라 스케일업이 더 나은 선택임을 확인했습니다.

**포인트**:

### Q6. 유사한 문제를 예방하기 위한 설계 원칙은?

**A**: 다음 설계 원칙을 적용해야 합니다:

1. **용량 계획(Capacity Planning)**: 예상 트래픽의 2-3배를 처리할 수 있는 인프라 설계. "1000명 동시 접속"이 목표라면 3000명까지 테스트.

2. **계층별 병목 분석**: 애플리케이션(BCrypt), 미들웨어(HikariCP), 데이터베이스(RDS) 각 레이어의 한계를 사전에 파악.

3. **워크로드 기반 인스턴스 선택**:
   - 웹 API: 범용(M5/M6i)
   - CPU 집약(암호화): 컴퓨트 최적화(C5/C6i)
   - 메모리 집약(캐싱): 메모리 최적화(R5/R6i)

4. **점진적 부하 증가 테스트**: 한 번에 목표 VU가 아닌, 단계적으로 증가시켜 병목 지점을 조기 발견.

5. **Rate Limiting 기본 적용**: 모든 인증 관련 API에 Rate Limiting 적용으로 DDoS/무차별 대입 공격 방어.

**포인트**:

- 레이어별 병목 분석 습관화
- 방어적 설계(Rate Limiting) 적용

## 핵심 교훈 (Key Takeaways)

### 1. BCrypt와 CPU 병목의 상관관계
- **문제**: BCrypt Cost Factor 10은 단일 검증에 100-200ms CPU 소요
- **교훈**: 보안과 성능은 트레이드오프 관계. 인프라 스케일업으로 보안 유지하며 성능 확보 가능
- **적용**: CPU 집약적 작업에는 Compute Optimized 인스턴스 사용

### 2. 부하 테스트의 중요성
- **문제**: 운영 전 부하 테스트 미실시로 인프라 한계 인지 못함
- **교훈**: 운영 배포 전 부하 테스트는 필수. 예상 트래픽의 2-3배까지 테스트
- **적용**: CI/CD 파이프라인에 성능 테스트 단계 추가 검토

### 3. AWS 인스턴스 타입 선택
- **문제**: T3(버스트)를 CPU 집약적 워크로드에 사용
- **교훈**: 워크로드 특성에 맞는 인스턴스 타입 선택 필수
- **적용**: 상시 고부하 → C5/C6i, 가변 부하 → Auto Scaling

## 11. HikariCP 커넥션 풀 분석

### 11.1 "무작정 커넥션 풀을 늘리면 안 된다"의 의미

커넥션 풀 사이징은 공식을 기반으로 신중하게 결정해야 합니다.

#### HikariCP 권장 공식

```
connections = (CPU cores × 2) + spindle count
```

- **CPU cores**: EC2 인스턴스의 vCPU 수
- **spindle count**: 디스크 스핀들 수 (SSD는 1로 계산)

#### 커넥션 수와 성능의 관계

```
┌─────────────────────────────────────────────────────────────────┐
│                  커넥션 수 vs 성능 그래프                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  처리량 │         ┌──────────── 최적점 (sweet spot)               │
│   (TPS) │        /│\                                             │
│         │       / │ \                                            │
│         │      /  │  \______ 성능 저하 시작                        │
│         │     /   │         (Lock 경합, Context Switching)       │
│         │    /    │                                              │
│         └───┴────────────────────────────────────────────        │
│              적정  과다         커넥션 수                          │
│                                                                  │
│  ⚠️ 과다 커넥션의 문제:                                           │
│  - DB 메모리 과다 사용 (커넥션당 ~5-10MB)                          │
│  - Lock 경합 증가                                                │
│  - Context Switching 오버헤드                                    │
│  - 결국 성능 저하                                                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 11.2 TPT-API 상황 분석

#### 인프라 사양

| 항목 | 현재 (업그레이드 전) | 변경 후 |
|------|---------------------|---------|
| EC2 | t3.medium (2 vCPU) | c6i.xlarge (4 vCPU) |
| RDS | db.t4g.medium (2 vCPU, 4GB) | db.r6g.large (2 vCPU, 16GB) |
| RDS max_connections | ~80 | **~660** |

#### 커넥션 풀 계산

**c6i.xlarge (4 vCPU) 기준:**

```
공식: (4 cores × 2) + 1 = 9 (이론값)
실무: 15-20 (BCrypt 같은 CPU-intensive 작업 고려)
```

#### 권장 설정

```yaml
# application.yml 권장 설정
spring:
  datasource:
    hikari:
      maximum-pool-size: 20      # 4 vCPU 기준 적정값
      minimum-idle: 8            # 평소 유지할 커넥션
      connection-timeout: 20000  # 20초 대기 후 예외
      idle-timeout: 300000       # 5분 유휴 후 제거
      max-lifetime: 1200000      # 20분 후 커넥션 갱신
      pool-name: TPT-HikariPool
```

## 핵심 교훈 (Key Takeaways)

### 1. BCrypt와 CPU 병목의 상관관계
- **문제**: BCrypt Cost Factor 10은 단일 검증에 100-200ms CPU 소요
- **교훈**: 보안과 성능은 트레이드오프 관계. 인프라 스케일업으로 보안 유지하며 성능 확보 가능
- **적용**: CPU 집약적 작업에는 Compute Optimized 인스턴스 사용

### 2. 부하 테스트의 중요성
- **문제**: 운영 전 부하 테스트 미실시로 인프라 한계 인지 못함
- **교훈**: 운영 배포 전 부하 테스트는 필수. 예상 트래픽의 2-3배까지 테스트
- **적용**: CI/CD 파이프라인에 성능 테스트 단계 추가 검토

### 3. AWS 인스턴스 타입 선택
- **문제**: T3(버스트)를 CPU 집약적 워크로드에 사용
- **교훈**: 워크로드 특성에 맞는 인스턴스 타입 선택 필수
- **적용**: 상시 고부하 → C5/C6i, 가변 부하 → Auto Scaling

### 4. 커넥션 풀 사이징의 과학
- **문제**: 커넥션 풀 사이징 근거 없이 기본값 사용
- **교훈**: HikariCP 공식 `(cores × 2) + 1` 기반으로 계산, 무작정 늘리면 역효과
- **적용**: EC2 vCPU 기준 15-20개 설정, RDS 한도 대비 여유 확보

### 5. Launch Template 기반 인프라 관리
- **문제**: EC2 직접 수정은 Blue/Green 배포 시 덮어씌워짐
- **교훈**: ASG + Launch Template 조합에서는 템플릿 버전 관리가 핵심
- **적용**: 스케일업 시 Launch Template 새 버전 생성 → ASG 업데이트
